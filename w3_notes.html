Object Literals are objects created by wrapping all of its properties and methods in braces that are mutable while the program is running, even if the object itself is a const.

Then there was the description of Symbols themseves was confusing because they used the same word "name" for the identifier, the description, and the property name.

It was unclear whether the property name was referencing the identifier or the description based on the example they gave. While this was confusing, it helped me see that even though I read about Symbols in chapter 2, I clearly didn't understand them well enough. I found a question on the motivation for bringing Symbols into ES6 on stackoverflow.com, which outlines the original purpose was to have private properties, but they were downgraded and are not private now. This didn't really help me much, so I kept searching. I found an article about Symbols in ES6 and why they're awesome. It talked about how Symbols are really just a different way to attach proprties to an object. They ensure a unique value. This still didn't click to me why you would need that- can't you just use unique strings? So I kept looking. I found another article titled JavaScript Symbols: But Why?. While reading through this one, which admittedly was saying the same thing again, I had an epiphany. My mind went to family history, and how I have 2 relatives names Charles Miles Estey (grandfather and grandson). FINALLY- This is why you might want a symbol! You cannot always control the names of things, and sometimes they might have the same string of characters, but are still unique, like my ancestors. This article talked about preventing name collisions when two libraries want to add properties and both happen to use the same key (which is apparently more common than I think it would be), symbols would be used to make each one unique.

I'm familiar with the DOM and various selector properties, but was interested to learn that when you pull node lists, they are array-like, but not arrays. More interesting was how easy it is to make it an array with Array.from(). However, even without converting it to an array, you can still do quite a few array-like things with the node list, like access each through index notation, get the length, or use the spread operator.

I was happy to be reminded of querySelector(), which I haven't used in quite some time. I like CSS, so using CSS notation is always good for me. Mental note- remember that it will only access the first element that matches the selector. This is useful for id's, as they are unique, so you'll always access what you're looking for. To find them all, use querySelectorAll(). You can use CSS pseudo-selectors with this, too.

I don't remember ever learning that many of the selector properties will return whitespace as text nodes. I found an interesting article where the author wrote some functions to test a node to see if it's ignorable to help get around the whitespace, but I think it would be more beneficial to understand how the whitespace is handled and recognized by the DOM and simply take my actions from there.

Another thing I appreciated being reminded of is that you need to use camel case when altering the CSS properties through JavaScript.

An important thing that I want to note is the importance of not calling things in a function that haven't loaded on the page yet. For the Quiz Ninja project, I had to move the script tag to the bottom of the page because it was starting the quiz when it loaded. With the script tag in the head of the page, it was trying to alter the innerHTML of elements before the elements had even loaded, causing an error.

Important info about event listeners: don't use the parentheses when the function is used as an argument to an event listener; can be used on a node or globally; pageX and pageY seem infinitely more useful than screenX, screenY, clientX, and clientY in locating coordinates as it relies on placement within the document itself, even accounting for scrolling; and refer often to the list of event listeners as there are more than you'd expect!

I need to consider when it would be more beneficial for events to capture instead of their native bubble, and when it might be useful to stop the even propagation all together.
